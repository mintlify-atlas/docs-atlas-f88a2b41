Components are the lowest layer of Eino. Each component is a Go interface with a small, focused contract. You write your application code against the interface; the concrete implementationā€”OpenAI, Ollama, a local vector store, or your own codeā€”is injected at construction time. Official implementations live in [eino-ext](https://github.com/cloudwego/eino-ext), and you can add your own by satisfying the interface.

## Component type identifiers

Every component category has a string constant in the `components` package. The framework uses these constants for callbacks, DevOps tooling, and display names.

```go
const (
    ComponentOfPrompt      Component = "ChatTemplate"
    ComponentOfChatModel   Component = "ChatModel"
    ComponentOfEmbedding   Component = "Embedding"
    ComponentOfIndexer     Component = "Indexer"
    ComponentOfRetriever   Component = "Retriever"
    ComponentOfLoader      Component = "Loader"
    ComponentOfTransformer Component = "DocumentTransformer"
    ComponentOfTool        Component = "Tool"
)
```

A component can also implement `Typer` to give itself a human-readable display name:

```go
type Typer interface {
    GetType() string
}
```

When `GetType()` returns `"OpenAI"` and the component is a `ChatModel`, the DevOps tooling shows it as `"OpenAIChatModel"`.

## Chat model

### BaseChatModel

`BaseChatModel` is the core interface for all chat model implementations. It supports two execution modes: blocking (`Generate`) and streaming (`Stream`).

```go
type BaseChatModel interface {
    Generate(ctx context.Context, input []*schema.Message, opts ...Option) (*schema.Message, error)
    Stream(ctx context.Context, input []*schema.Message, opts ...Option) (
        *schema.StreamReader[*schema.Message], error)
}
```

The input is a slice of `*schema.Message` representing the conversation history. Messages carry a role (`system`, `user`, `assistant`, `tool`) and support multimodal content. When reading a `StreamReader`, close it when you are done and read it only once ā€” copy it first if multiple consumers need the same stream.

### ToolCallingChatModel

`ToolCallingChatModel` extends `BaseChatModel` with concurrency-safe tool binding. Unlike the deprecated `ChatModel.BindTools`, `WithTools` returns a new instance and does not mutate the receiver.

```go
type ToolCallingChatModel interface {
    BaseChatModel

    // WithTools returns a new ToolCallingChatModel with the specified tools bound.
    // Safe for concurrent use ā€” does not modify the receiver.
    WithTools(tools []*schema.ToolInfo) (ToolCallingChatModel, error)
}
```

You can share a base model instance across goroutines and derive per-request variants:

```go
base, _ := openai.NewChatModel(ctx, cfg)           // shared, no tools
withSearch, _ := base.WithTools([]*schema.ToolInfo{searchTool})
withCalc, _   := base.WithTools([]*schema.ToolInfo{calcTool})
```

### Using a chat model

```go
chatModel, _ := openai.NewChatModel(ctx, &openai.ChatModelConfig{
    Model:  "gpt-4o",
    APIKey: os.Getenv("OPENAI_API_KEY"),
})

messages := []*schema.Message{
    schema.SystemMessage("You are a helpful assistant."),
    schema.UserMessage("What is the capital of France?"),
}

response, _ := chatModel.Generate(ctx, messages)
fmt.Println(response.Content)
```

## Tool

Tools give a chat model the ability to take actions. There are two core interfaces:

### BaseTool

`BaseTool` provides the metadata a chat model uses to decide whether and how to call a tool. `Info` returns a `*schema.ToolInfo` containing the tool name, description, and parameter JSON schema.

```go
type BaseTool interface {
    Info(ctx context.Context) (*schema.ToolInfo, error)
}
```

`BaseTool` alone is sufficient when passing tool definitions to a model via `WithTools`. To also execute the tool, implement `InvokableTool` or `StreamableTool`.

### InvokableTool

`InvokableTool` is a tool that `ToolsNode` can execute. It receives the model's tool-call arguments as a JSON-encoded string and returns a plain string result sent back to the model as a tool message.

```go
type InvokableTool interface {
    BaseTool

    InvokableRun(ctx context.Context, argumentsInJSON string, opts ...Option) (string, error)
}
```

### StreamableTool

`StreamableTool` is the streaming variant of `InvokableTool`. It returns a `*schema.StreamReader[string]` that yields string chunks incrementally.

```go
type StreamableTool interface {
    BaseTool

    StreamableRun(ctx context.Context, argumentsInJSON string, opts ...Option) (*schema.StreamReader[string], error)
}
```

### EnhancedInvokableTool

`EnhancedInvokableTool` supports structured multimodal results. Arguments arrive as `*schema.ToolArgument` and the result is `*schema.ToolResult`, which can carry text, images, audio, video, and file content. When a tool implements both a standard and an enhanced interface, `ToolsNode` prioritises the enhanced one.

```go
type EnhancedInvokableTool interface {
    BaseTool
    InvokableRun(ctx context.Context, toolArgument *schema.ToolArgument, opts ...Option) (*schema.ToolResult, error)
}
```

## Retriever

A `Retriever` fetches the most relevant documents from a store for a natural-language query.

```go
type Retriever interface {
    Retrieve(ctx context.Context, query string, opts ...Option) ([]*schema.Document, error)
}
```

Options include `WithTopK` to cap the number of results and `WithScoreThreshold` to filter out low-relevance documents. When `WithEmbedding` is set, the retriever converts the query to a vector before searching ā€” use the same embedding model that was used at index time.

```go
docs, _ := retriever.Retrieve(ctx, "what is eino?", retriever.WithTopK(5))
```

## Embedder

An `Embedder` converts a batch of strings into dense vector representations. The vector length is fixed by the underlying model and identical for every text in the batch.

```go
type Embedder interface {
    EmbedStrings(ctx context.Context, texts []string, opts ...Option) ([][]float64, error)
}
```

Both `Indexer` and `Retriever` use an `Embedder`. They must share the exact same model ā€” mismatched dimensions or model families break semantic similarity.

## ChatTemplate (prompt)

A `ChatTemplate` formats a variables map into a list of messages for a chat model.

```go
type ChatTemplate interface {
    Format(ctx context.Context, vs map[string]any, opts ...Option) ([]*schema.Message, error)
}
```

The substitution syntax (FString, GoTemplate, or Jinja2) is determined at construction time. In a graph or chain, `ChatTemplate` typically precedes `ChatModel`. Variable keys present in the template but absent from the map produce a runtime error.
