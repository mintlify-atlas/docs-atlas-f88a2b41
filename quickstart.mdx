description: "Install the Eino Go module, connect a ChatModel backed by OpenAI, create a ChatModelAgent with the ADK, and run your first query in a few minutes."

In this guide you will install Eino, configure a ChatModel backed by OpenAI, create a `ChatModelAgent` that handles the ReAct loop for you, and run a query against it. You will also see how to add tools to your agent and how to build a composed graph for when you need precise control over execution flow.

<Steps>

<Step title="Install Eino">

Add the core Eino module to your project:

```bash
go get github.com/cloudwego/eino
```

You also need an implementation for the model provider you want to use. For OpenAI:

```bash
go get github.com/cloudwego/eino-ext/components/model/openai
```

See [Installing Eino and its dependencies](/installation) for a full list of available implementations.

</Step>

<Step title="Configure a ChatModel">

Create a `ChatModel` using the OpenAI implementation. The model is the bridge between Eino and the LLM API.

```go
import (
    "context"
    "os"

    "github.com/cloudwego/eino-ext/components/model/openai"
)

ctx := context.Background()

chatModel, err := openai.NewChatModel(ctx, &openai.ChatModelConfig{
    Model:  "gpt-4o",
    APIKey: os.Getenv("OPENAI_API_KEY"),
})
if err != nil {
    // handle error
}
```

`ChatModel` is an interface defined in the core `eino` module. Swapping providers — for example, switching from OpenAI to Ollama — requires only changing this initialization call.

</Step>

<Step title="Create an agent">

Pass the `ChatModel` to `adk.NewChatModelAgent`. The agent handles the ReAct loop internally: it decides when to call tools and when to return a response.

```go
import "github.com/cloudwego/eino/adk"

agent, err := adk.NewChatModelAgent(ctx, &adk.ChatModelAgentConfig{
    Model: chatModel,
})
if err != nil {
    // handle error
}
```

</Step>

<Step title="Run queries with a Runner">

`adk.NewRunner` wraps an agent and provides the `Query` convenience method. It returns an `AsyncIterator` that yields `AgentEvent` values as the model streams its response.

```go
runner := adk.NewRunner(ctx, adk.RunnerConfig{Agent: agent})

iter := runner.Query(ctx, "Hello, who are you?")
for {
    event, ok := iter.Next()
    if !ok {
        break
    }
    if event.Err != nil {
        log.Printf("error: %v", event.Err)
        continue
    }
    if event.Output != nil && event.Output.MessageOutput != nil {
        msg, err := event.Output.MessageOutput.GetMessage()
        if err == nil {
            fmt.Println(msg.Content)
        }
    }
}
```

</Step>

</Steps>

## Add tools to your agent

Give the agent capabilities by passing tools through `ToolsConfig`. The agent automatically receives tool descriptions, decides when to call each tool, and loops until it has a final answer.

```go
import (
    "github.com/cloudwego/eino/adk"
    "github.com/cloudwego/eino/components/tool"
    "github.com/cloudwego/eino/compose"
)

agent, err := adk.NewChatModelAgent(ctx, &adk.ChatModelAgentConfig{
    Model: chatModel,
    ToolsConfig: adk.ToolsConfig{
        ToolsNodeConfig: compose.ToolsNodeConfig{
            Tools: []tool.BaseTool{weatherTool, calculatorTool},
        },
    },
})
```

Any value that implements `tool.BaseTool` can be passed here. See the [eino-ext](https://github.com/cloudwego/eino-ext) repository for ready-made tool implementations, or implement the interface yourself.

## Build a composed graph

When you need precise control over execution flow — fixed ordering, conditional branching, or parallel steps — use `compose.NewGraph` instead of the ADK agent pattern.

```go
import "github.com/cloudwego/eino/compose"

graph := compose.NewGraph[*Input, *Output]()
graph.AddLambdaNode("validate", validateFn)
graph.AddChatModelNode("generate", chatModel)
graph.AddLambdaNode("format", formatFn)

graph.AddEdge(compose.START, "validate")
graph.AddEdge("validate", "generate")
graph.AddEdge("generate", "format")
graph.AddEdge("format", compose.END)

runnable, err := graph.Compile(ctx)
if err != nil {
    // handle error
}

result, err := runnable.Invoke(ctx, input)
```

Graphs are typed: the compiler checks that each edge's output type matches the next node's input type. You can also expose a compiled graph as a tool and pass it to an agent, letting the agent decide when to invoke your deterministic pipeline.

</Tip>
